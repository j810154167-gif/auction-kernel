# 先读这个

## 仓库警告

这个仓库里的多数文件是 Jackson 和 Hermes Agent 之间内部工作材料——工单、测试报告、历史产物。**不要被它们带偏。**

以下是 fable5 需要关注的唯一路径：

| 优先级 | 文件 | 用途 |
|--------|------|------|
| **必读** | `PROBLEM.md` | 困境陈述：问题是什么、为什么两个内核都失败了、结构死锁在哪 |
| 参考 | `../kernel.py` | 新内核当前代码（知道它在做什么，不需要改进它） |
| 参考 | `../README.md` | 设计约束（闸门禁令、阈值禁令、排序禁令） |

其他全部文件可以忽略。

## 核心约束（从 Jackson 直接来）

1. **这是对手盘博弈问题，不是数据组织问题。** 如果你给的答案可以用"加一个字段""调一个权重""换一种分区方式"来描述，那就是错的。
2. **不要排序、不要闸门、不要阈值。** 这些已经反复证明是过拟合。
3. **机器的边界是呈现事实。** 判断永远归 Jackson。但呈现方式必须让 Jackson 在五分钟内可决策——不能让他当场发明排序。
4. **每天都是新的买入。** 没有持仓惯性、没有成本锚定。

## 策略框架 — 两条被遗漏的前提

**涨停就是天然广告。** 5000只票的数据洪流对人脑无意义。市场自身的涨停机制说了"今天看这里"——Jackson的策略不是从5000只里找信号，是从市场选出的涨停票中，在次日竞价判断真/假资金承诺。

**进攻和防守是同一个循环。** T-1涨停池 → T日竞价入场 → 防守监控持仓 → T日收盘新涨停进入候选。两个内核都把进攻当成独立排序事件，但 Jackson 要的是一个以涨停为锚点的持续循环。

见 `PROBLEM.md` 中「被遗漏的策略框架」章节的完整陈述。

## 不要做的

- 不要优化 kernel.py
- 不要提议新的数据字段或新的 API
- 不要给"更好的分区公式"或"更科学的权重"
- 不要分析 0720 对比数据来"找到规律"——0720 的结论已经在了：数据证明不了什么，只是证明了死锁存在
