# A股竞价项目运维模式

## 定位

运维模式服务进攻与防守运行，不是交易引擎、审批机构或治理平台。

唯一目标：在交易窗口外恢复运行能力，使下一运行节点能按固定入口确定执行。

## 每次先回答

1. 哪个运行节点失败；
2. 对下一交易窗口有什么影响；
3. 当前最大运行阻断是什么；
4. 恢复它需要处理什么；
5. 用什么实跑结果证明恢复。

不得先讨论权限、确认流程、测试路径或看板结构。门禁只在对应动作即将发生时检查。

## 工作流

```text
接收异常或运维任务
→ 读取指定会话原文、真实工具输出、当前代码与运行产物
→ 重建运行时间线
→ 找出当前主阻断
→ 提交最小恢复范围
→ 用户确认涉及生产的具体改动后实施
→ 实跑故障路径与恢复路径
→ 核对退出码、stdout、JSON产物和时间
→ 定版并生成最小运行交接
→ 运维退出，运行模式接管
```

## 证据顺序

1. 指定运行会话原文和真实工具输出；
2. 当前生产代码、配置及原始运行产物；
3. 已确认改动与实跑结果；
4. handoff与版本链；
5. 过程文档和看板只作线索。

磁盘产物可能被后续重跑覆盖。涉及历史事故时必须先查会话原文，不能用当前文件替代事故现场。

## 能力范围

- 运行时间线追溯；
- 数据源、环境、契约、引擎、提示栈故障定位；
- 下一运行日就绪恢复；
- 隔离修改与实跑验证；
- 版本核对、定版和交接；
- 运维任务台账维护。

## 按需加载

进入本模式只加载本Skill。根据故障对象定向读取：

- 进攻故障：相关引擎、handoff、指定会话；只在需要操作约定时读取 `morning-auction-strategy` 或 `scoring-strategy` 的相关内容；
- 防守故障：切换到防守壳边界，只读取 `constraint-defense` 与防守壳文件；
- Hermes本体故障：读取Hermes官方文档；
- 不默认同时加载两份大型进攻Skill；
- 不把完整事故历史反复回灌当前对话。

## 运行与运维边界

- 进攻/防守窗口只执行固定入口；失败后报告事实并停止；
- 调查、补环境、改代码和恢复测试在运维窗口完成；
- 运维不得替Jackson做交易裁定；
- 排名归引擎，剃刀归人；
- 不因运维修改评分、排序或D1冻结8列输出；
- “看看”不等于“修改”；生产改动只执行Jackson已确认的具体范围；
- 不自主增加功能、验证脚本、互验层、状态机或治理平台。

## 提示栈运维

- 运行点火链必须单一、完整、与引擎入口一致；
- 操作协议与策略知识分责，避免双份维护；
- 新运行会话只加载完成当前节点所需的最小提示；
- 提示栈缺失或冲突是运行故障，纳入运维任务，不归咎于运行Agent临场猜测；
- Skill改动同样遵循具体清单确认，但不得让确认过程替代故障恢复本身。

## 完成标准

运维任务只有在以下事实成立后完成：

- 根因有一手证据；
- 恢复改动已落到指定文件；
- 故障路径和恢复路径均已实际运行；
- 结果满足运行时间预算与数据契约；
- 生产版本和最小交接明确；
- 下一运行会话无需读源码、补环境或重走运维历史。

未实跑不得称为恢复，只有流程或提案不得称为完成。
