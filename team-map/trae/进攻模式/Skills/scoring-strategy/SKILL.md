# Scoring 策略 — 人脑决策友好

## ⛔ Agent 行为协议（不可违反）

以下规则与 `morning-auction-strategy` 协议联动。违反任一条视为故障。

- **D1 终端输出格式冻结**。未经用户明确指令，禁止修改 D1 输出行格式、添加/删除字段。
- **先提案、等确认、再编码**。任何涉及评分公式、排序逻辑、输出格式的变更，必须先列变更清单→用户确认→执行。
- **禁止自主优化**。"看看"≠"改"。
- **标记密度上限**。D1 每行最多 3 个 marks 词。
- **输出格式修改流程**：先仿写原文 → 用户对照确认 → 再编码固化。禁止跳过仿写直接改。0714教训：跳过仿写直接改格式导致55轮无效修改。

## 运行协议职责

进攻模式的点火词、时间链、盘后闭环和Agent执行约定，以 `morning-auction-strategy` 为唯一操作协议。本Skill只定义评分、排序、管线标记及其数据知识，不复制运行入口。

## 架构定位

| | 约束系统（防守） | Scoring（进攻） |
|------|------|------|
| 决策者 | Agent 机器脑 | 人类交易者 |
| 输出 | Gate标记池(无排序) | 对照表(管线标记+参照物) |
| 过滤 | 做减法(砍高开) | 做压缩(5000→漏斗) |
| 偏好 | 保守(空仓防御) | 激进(人做最后一刀) |

**注(0714)**: 0618原设计是三闸对照（价vsVWAP、量vs昨日、梯队vs板块），不是chg×3评分。chg×3/红线/竞涨幅排序——全部被拒绝为参数化过拟合。D1当前输出为对照表+管线标记。博弈检测 > 评分排序。

## 与 constraint-defense 的关系

Scoring 是进攻端，constraint-defense 是防守端。双壳架构，物理隔离：

| | 进攻(scoring) | 防守(constraint-defense) |
|------|------|------|
| 时区 | 盘中 09:15-09:35 | 盘后 + 盘中静默监控 |
| 输出 | ★1~N排序 | 🔴卖出/🟡预警 |
| 决策权 | 你选+你入 | 你确认卖/不卖 |
| 共享 | → handoff/entry.json | ← handoff/entry.json |
| 心智 | 独立壳，不混记忆 | 独立壳，不混记忆 |

入场后防守Agent即时建档开始监控。"每天都是新的买入"——防守无成本执念，只看区间是否被破。

## 独立建池流程

1. TickFlow `/v1/exchanges/{SH,SZ}/instruments` 实时拉取 → 过滤：主板(60/00/002/003) + **排除 688/300/301/8/4/ST**
   - 科创板/创业板 20cm 涨跌幅会放大 chg×3，必须排除
2. TickFlow klines/batch count=10 → 识别 T-1 涨停(≥9.5%)
3. 5日 VWAP + floor(×0.85) → 候选池
4. 当日竞价 open → chg 排序 (无乘法系数)
5. ★1~N + 六模块管线标记 + L0 生命线
6. 09:15 起允许启动行情采集 / WS；执行 auction 后必须检查 `auction_snapshot.json` 的 `meta.mode`、open=0 数量、volume=0 数量。
7. 09:25 后 D1 读取沉淀后的竞价快照，人类五分钟窗内逐只看盘面做最后一刀

### 人类决策链（不是全自动跟单）

```
5000只全量 → 算法扫到74 → 评分排到10 → 人类盘感剃 → 最终买N只
                                    ↑
                              这刀只有人类能做
```

评分系统的 ~50% 胜率不是缺陷——它是给人留的决策空间。算法完成的是"压缩范围"，不是"做买卖决策"。

### 0713 回测验证

详见 `references/0713-calibration.md`。

三版对比，验证了过滤规则的必要性：

| 版本 | 池 | Top10胜率 | 说明 |
|------|:--:|:---:|------|
| 混池(v1) | 约束系统25只 | — | 池不对，全负分 |
| 全市场(v2) | 含科创/创业 122只 | 10% | 20cm放大chg×3 |

### 🔴 chg×3 + max(0,dev-7)×2 = 断崖根因 (0714发现)

`×3` 放大竞涨幅，`×2` 放大偏差惩罚。两者互斥（高chg天然高dev），修完 close_t1 数据修正后公式崩盘：

```
修正前(prev_close=T-2, 数据虚涨):
  chg分布 12~21% → ×3后36~63 → dev-7惩罚被大基数淹没 → 平滑梯度 → 哈药/金开成功不是公式好

修正后(close_t1=昨收, 数据正确):  
  chg分布 2~9% → ×3后6~27 → dev-7惩罚主导(大部分dev>7扣8~16分) → ★1(33.3)到★2(7.7)断崖25.6分
```

**×3 ×2 没有市场逻辑**——是0625当天试出来顺眼的数。数据修正后暴露了公式的结构缺陷。去掉×3×2后断崖消失(15.8→4.0平滑)。0713哈药/金开不是公式好，是prev_close bug让所有chg虚高淹没了惩罚项。

## 评分公式 (当前: ×3×2已移除)

```
score = chg - max(0, dev-7) + min(vol/10000, 5) + vol_bonus - penalty×0.4
```

Gate罚分: ①价(破底/超天花板) ②量(无竞价-8) ③梯队(chg≤0-10) ①b(偏差>P75)

## Constraint 六层管线标记 — JSON 仅存

管线在 `node_d1()` 中执行（in-place 加 marks），不改变分数排序。标记仅存于 `decision_1.json`，**不输出到 D1 终端**——D1 终端仅显示分数 + L0 生命线。需要时再展开 JSON。

## 数据陷阱

1. **change_pct 双重乘100**：`auction_snapshot.json` 中 `change_pct` 已被引擎 ×100 处理过（如 0.00985 → 0.985）。回测代码若再 `if abs < 1: *100` 会导致 +98.5% 虚高。验证：用 `(last_price - prev_close)/prev_close` 反推真实涨跌。正确做法是直接用 D1 的 `decision_1.json` 中的 `change_pct` 字段（已正确）。
2. **分钟K date参数无效**：`/v1/klines?date=2026-07-10` 被忽略，返回当日。必须用 `start_time`/`end_time` 毫秒时间戳。不用 `/v1/klines/intraday`（只返回今日）。
3. **科创板/创业板 20cm 必须排除**：688/300/301 的 20% 涨跌幅放大 chg×3。0713回测验证：Top10中6/10是20cm标的，胜率仅10%。主板过滤后胜率回升至50%。
4. **竞价量不可用 post-auction volume**：回测时拉到的 `volume` 已是连续竞价量，非竞价量。gate_penalty 统一施加"无竞价数据(-8)"不影响排序，但不要据此做量能判断。
5. **标的列表实时获取**：通过 TickFlow `/v1/exchanges/SH/instruments` + `/v1/exchanges/SZ/instruments` 实时拉取，不再依赖过期 CSV。过滤: 排除 ST、688(科创)、300/301(创业)、8/4(北交)、000(SH指数)。
6. **K线棒索引漂移**：`klines/batch` 返回"最近 N 根棒"，日期位置随调用时间漂移。不能用 `closes[3]` 固定索引——次日调用时棒位全部偏移。必须解码 `timestamp` 字段建 `date_idx`，按日期查 T-1/T-2。同时 `count=5` 不足（盘后/周末可能覆盖不到目标日），改用 `count=10`。
7. **close_t1 vs prev_close**：池里的 `prev_close` 是 T-2 收盘（建池用），不是昨日收盘。`open_chg_pct` 必须用 `close_t1`（昨收）。误用导致竞价涨幅虚高（水发燃气 0714: +19.7%→+8.8%）。两文件已修: `auction_engine.py node_d1()` + `scoring_engine.py fetch_pool_quotes()`。

## 市场规则协议

链式校验参照物。主板 ±10% 是硬约束——机器不判断"对不对"，只判断"可不可能"。容忍上限含价格四舍五入。

| 板块 | 代码前缀 | 涨跌幅 | 容忍上限 |
|------|------|:--:|:--:|
| 主板 | 60/00/002/003 | ±10% | **11.0%** |
| 科创板 | 688 | ±20% | 21.0% |
| 创业板 | 300/301 | ±20% | 21.0% |
| 北交所 | 8/9/4 | ±30% | 31.5% |
| ST | — | ±10% | 10.5% |

特殊: 新股首日无限制(不在池中), 除权日可能超限(标记"可能除权"不阻断)。

## 链式校验

`node_startup()` 启动时执行: MARKET_RULES(前置)→⓪时间校验→①数据源→②数据校验(违规自动重建池)→③版本→PROCEED/BLOCKED。串联阻断。

## WS 主链路

09:15 是竞价窗口起点。09:15 起允许启动行情采集 / WS；09:25 后 D1 读取沉淀后的竞价快照。

竞价采集 WS 为 D1 前置条件。WS 不可用则 D1 不启动——人知道今天不做。REST 降级为 WS 故障时的辅助。若 `auction_snapshot.json` 产物 `meta.mode=rest`，必须按事实说明本轮不是 WS 常驻流，不得称为 WS 已启动。执行 auction 后必须上报 `meta.mode`、quotes 数量、open=0 数量、volume=0 数量。

## L0 生命线标记 (0714 v2)

D1 得分后一列。昨日 VWAP 基准，三级判定：

| 条件 | 标记 | 含义 |
|------|:--:|------|
| 竞 < VWAP | 🔴 | 破线——做市商放手，昨全体浮亏 |
| 竞 > VWAP×1.15 | 🔴 | 极端溢价——竞价虚高，陷阱风险 |
| 竞 > VWAP×1.10 | 🟡 | 高溢价——谨慎 |
| 其余 | 🟢 | 正常——竞在平均成本线内 |

代码位置: `auction_engine.py` L0 生命线标记段，`p < v or p > v * 1.15 → 🔴 | p > v * 1.10 → 🟡 | else 🟢`。

## 输出渲染陷阱 (0714 教训)

**引擎 `print()` 输出到 `stdout` 的 markdown 表格 vs Agent 回复消息中渲染的 markdown 表格——格式相同但 UI 表现不同。** 终端 `stdout` 可能被截断或不渲染表格。确认 D1 输出正确性的唯一方法：Agent 读取 `stdout` 后直接在回复中渲染 markdown。禁止仅凭终端预览判断格式。
