## 一、项目数据需求矩阵

### 1.1 按数据维度拆分


| 数据维度        | 当前源               | 协议        | 频率        | 覆盖范围        | 必需字段                                                           | P0? |
| ----------- | ----------------- | --------- | --------- | ----------- | -------------------------------------------------------------- | --- |
| **实时报价**    | TickFlow WS       | WebSocket | 实时 tick   | 竞价池 ≤100只   | symbol, last_price, open, volume, amount, change_pct           | 🔴  |
| **日K线**     | TickFlow REST     | HTTP GET  | 盘前一次      | 全A股 ~3194只  | close[2], volume[2], amount[2], high[2], low[2]                | 🔴  |
| **1分钟K线**   | TickFlow REST     | HTTP GET  | 盘中按需      | D2候选池 ≤100只 | open[], close[], volume[], timestamp[]                         | 🟡  |
| **集合竞价**    | TickFlow WS 间接    | WebSocket | 9:15-9:25 | 竞价池         | 隐含在实时quote中 (open, volume)                                     | 🔴  |
| **行业/概念板块** | 东方财富 push2        | HTTP GET  | 盘前+盘中     | Top10行业+概念  | code, name, change_pct, turnover, up/down_count, leading_stock | 🟡  |
| **板块成分股**   | 东方财富 push2        | HTTP GET  | 盘中        | 每个板块≤200只   | f12 (股票代码)                                                     | 🟡  |
| **涨停原因/概念** | i问财 SkillHub      | MCP间接     | T-1快照     | 涨停股 ~170只   | 涨停原因, 所属概念, 连续涨停天数, 封流比, 封成比                                   | 🟡  |
| **股票码表**    | 本地 CSV            | 静态文件      | —         | 全A股主板       | code (sh.60xxxx / sz.00xxxx), code_name                        | 🟢  |
| **交易日历**    | cn-stock-holidays | PyPI包     | —         | A股          | is_trading_day                                                 | 🟢  |


### 1.2 关键约束


| 约束           | 说明                                 |
| ------------ | ---------------------------------- |
| WebSocket 必须 | 实时报价需要 WS 推送，REST 轮询无法满足竞价时段的毫秒级需求 |
| 竞价时段特殊       | 9:15-9:25 集合竞价阶段，普通行情源可能不覆盖        |
| 全市场扫描        | 日K线预加载需要一次拉取 ~3000+ 只股票的 T-1 数据    |
| 批量查询         | 需要单次请求支持多标的（batch query），逐个查询不可行   |
| 板块映射         | 需要 行业+概念 的双层分类体系，用于涨停原因推断          |

### 为什么选这个组合？


| 决策点             | 理由                                                    |
| --------------- | ----------------------------------------------------- |
| **黑狼数据为主力**     | 专为A股量化设计，53字段实时行情、竞价接口、资金流向全覆盖，是 TickFlow 的最直接替代      |
| **AKShare 补板块** | 黑狼数据缺行业/概念板块分类体系，AKShare 完美覆盖东方财富+同花顺双源板块数据           |
| **保留 i问财**      | 涨停原因解释层已有成熟 SkillHub 通路，替换成本高、收益低                     |
| **不选 iTick**    | 虽然 WS+REST 双协议、免费套餐诱人，但 A股竞价数据和板块分类支持弱，且专业版 $50/月超出预算 |
| **不选 XTick**    | 功能最匹配（竞价实时+WS全推），但价格未公开、Java SDK 为主，Python 生态弱        |
| **不选 AllTick**  | $99/月起超出预算，且 A股覆盖弱（侧重美股港股）                            |


### ⚠️ 关键风险：WebSocket 缺失

黑狼数据和 AKShare **都不提供 WebSocket 实时推送**，这意味着：


| 影响模块                 | 当前方式         | 替代方式             | 风险                |
| -------------------- | ------------ | ---------------- | ----------------- |
| `auction_monitor.py` | WS 实时推送 tick | HTTP 轮询 (1-3秒间隔) | 竞价时段延迟增加，可能错过极短窗口 |
| `d2_engine.py`       | WS 订阅持续流     | HTTP 按需拉取        | 分钟级 VWAP 累积精度下降   |


**缓解措施**：

1. 竞价时段 (9:15-9:25) 提高轮询频率至 500ms
2. 如果轮询延迟不可接受，可单独补充 **iTick 免费版 WebSocket**（仅用于实时报价推送，不付费）

## 六、迁移路径建议

### Phase 1: 影子运行 (1-2天，零风险)

```
1. 部署 fxyz_adapter.py → 静默采集，不接入 DAG
2. 部署 akshare_adapter.py → 静默采集板块数据
3. 输出 shadow_handoff/ 对比报告 → 与 TickFlow 数据交叉验证
```

### Phase 2: 降级链路 (1天)

```
4. 在 data_source_governance.py 注册黑狼数据为 Tier-2 降级源
5. 在 rest_recovery.py 添加 fxyz fallback 通路
6. 保持 TickFlow 为 Tier-1 主源
```

### Phase 3: 主源切换 (仅当 TickFlow 不可用时)

```
7. 提升黑狼数据为 Tier-1
8. 调整 auction_monitor.py 轮询模式
9. 全量回归测试 (tests/ 22个用例)
```

## 七、结论


| 问题               | 答案                                                             |
| ---------------- | -------------------------------------------------------------- |
| 能找到完全免费的优质替代吗？   | ❌ 不能。免费源要么无WS实时推送，要么无竞价数据，要么无板块分类。三者无法兼得。                      |
| ¥100/月内最优解？      | 🥇 **黑狼数据 + AKShare**，¥30-80/月，覆盖 90% 需求，唯一缺口是 WebSocket 实时推送。 |
| WebSocket 缺口怎么办？ | ① 提高 HTTP 轮询频率缓解；② 单独挂 iTick 免费版 WS 做实时报价补充。                   |
| 竞价数据怎么办？         | 黑狼数据有专门的集合竞价接口（9:15-9:25），是目前低价方案中最完善的。                        |
| 建议立即迁移吗？         | ⚠️ 不建议。TickFlow 当前可用，应先做影子运行验证数据质量，再逐步建立降级链路。                  |
